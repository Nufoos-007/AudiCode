## 1. Project Overview

AudiCode is an AI-powered security vulnerability scanner designed specifically 
for "vibe-coded" (AI-generated) applications built with tools like Cursor, 
GitHub Copilot, Bolt, Lovable, and similar AI coding assistants.

Mission: Make AI-generated code secure by automating security audits with 
real-time async scanning, confidence scoring, and actionable fixes.

Key Stats:
- 62% of AI-generated code contains vulnerabilities (CSA/Endor Labs 2025)
- 45% fails OWASP Top 10 (Veracode 2025)
- AI-generated code has 2.74x more vulnerabilities than human-written

## 3. Architecture

### System Flow (Current - Synchronous):

```
User Input ──▶ /api/scan ──▶ GitHub API ──▶ Scan ──▶ Results
                     (inline)   (REST)   (regex)
```

### System Flow (Async - Current Implementation):

```
┌──────────────────┐     ┌─────────────────┐     ┌──────────────────┐
│   Frontend       │     │  /api/jobs      │     │  Background     │
│  (Landing)       │────▶│  (Job Queue)    │────▶│  Worker         │
│  (Dashboard)     │     │  returns job ID │     │  (Async Scan)  │
│  (Auth)         │     └─────────────────┘     └──────────────────┘
│                  │            │                        │
│                  │            ▼                        ▼
│                  │     ┌─────────────────┐     ┌──────────────────┐
│                  │     │  In-Memory     │     │  GitHub API     │
│                  │     │  Jobs Store    │     │  (Files + CVE) │
│                  │     └─────────────────┘     └──────────────────┘
└──────────────────┘                                    │
                                                     ▼
                                             ┌──────────────────┐
                                             │  Store Results  │
                                             │  + Confidence  │
                                             └──────────────────┘
```

### Authentication Flow:
1. User visits https://audi-code.vercel.app
2. Clicks "Get Started" → /auth page
3. Chooses GitHub or Google OAuth (Supabase handles)
4. After login, redirected to /dashboard
5. Can select from their GitHub repos or enter URL

### Scan Flow (Async):
1. User selects repo to audit
2. POST /api/jobs with repoUrl
3. Returns jobId immediately (202 response)
4. Background worker:
   - Fetches repo via GitHub API (recursive)
   - Scans files against 50+ OWASP rules
   - Checks dependencies against GitHub Advisory
   - Calculates confidence scores
5. User polls GET /api/jobs?jobId=XXX
6. Returns complete results when done

### Scan States:
- queued: Job created, waiting to start
- running: Currently scanning files
- completed: Done with full results
- failed: Error occurred

## 4. Scanner Implementation

### Detection Method: Regex Pattern Matching

The scanner uses regex patterns to detect known vulnerability signatures:

```typescript
// Example rules from api/scan.ts
const OWASP_RULES = [
  // SQL Injection
  { pattern: /execute\s*\(\s*["'][^"']*\${\s*\w+\s*["']/g, 
    severity: "critical", title: "SQL Injection" },
  
  // Hardcoded Secrets  
  { pattern: /sk-[a-zA-Z0-9]{48}/g,
    severity: "critical", title: "OpenAI Key Exposed" },
    
  // Weak Crypto
  { pattern: /crypto\.createHash\s*\(\s*["']md5["']\)/g,
    severity: "high", title: "Weak Hash (MD5)" },
    
  // XSS
  { pattern: /\.dangerouslySetInnerHTML\s*=/g,
    severity: "high", title: "React XSS Risk" },
    
  // Command Injection
  { pattern: /eval\s*\(/g,
    severity: "critical", title: "Dangerous eval()" },
    
  // ... 50+ more rules
];
```

### File Filtering
- Skips test files (.test., .spec., .mock.)
- Skips node_modules
- Skips build/dist folders
- Skips scanner's own code (api/scan.ts)
- Limits to files < 40KB per file

### Async Job Processing
- Returns job ID immediately (202 status)
- Background worker processes scan
- In-memory job storage (production: Supabase)
- Polling endpoint for results

## 5. Vulnerability Categories (OWASP Top 10 + Vibe-Specific)

### A01: Broken Access Control
| Pattern | Severity | Description |
|---------|----------|-------------|
| Unprotected routes | critical | No auth middleware |
| Missing RLS | high | Supabase without policies |

### A02: Cryptographic Failures  
| Pattern | Severity | Description |
|---------|----------|-------------|
| Hardcoded API keys | critical | Keys in source |
| Weak crypto (MD5/SHA1) | high | Broken algorithms |
| Math.random() for tokens | high | Not cryptographically secure |

### A03: Injection
| Pattern | Severity | Description |
|---------|----------|-------------|
| eval() usage | critical | Arbitrary code execution |
| exec() usage | critical | Shell commands |
| SQL injection (templates) | critical | String interpolation in SQL |
| NoSQL injection | critical | MongoDB query injection |
| Command injection (shell=True) | critical | Shell=True vulnerability |

### A04: Insecure Design
| Pattern | Severity | Description |
|---------|----------|-------------|
| No rate limiting | medium | Missing rate limits |

### A05: Security Misconfiguration
| Pattern | Severity | Description |
|---------|----------|-------------|
| Wildcard CORS (*) | high | Allow all origins |
| DEBUG=True | critical | Debug in production |

### A06: Vulnerable Components
| Pattern | Severity | Description |
|---------|----------|-------------|
| CVE in dependencies | high | Known vulnerabilities |

### A07: Authentication Failures
| Pattern | Severity | Description |
|---------|----------|-------------|
| Hardcoded JWT secrets | high | Secrets in code |
| Weak session config | medium | Poor session handling |

### A08: Software & Data Integrity
| Pattern | Severity | Description |
|---------|----------|-------------|
| Git dependencies | medium | Untrusted sources |

### A09: Logging Failures
| Pattern | Severity | Description |
|---------|----------|-------------|

### A10: Server-Side Request Forgery (SSRF)
| Pattern | Severity | Description |

### Additional (Vibe-Coder Specific)

## 6. Confidence Scoring System

Every finding includes a confidence score (40-100):

### Confidence Levels:
- **95-100**: Very likely real - direct pattern match in code
- **70-94**: Likely real - pattern with context
- **40-69**: Uncertain - review manually

### Factors That Increase Confidence:
- Variable assignments (`const x = ...`)
- Function calls in the line
- More files scanned = higher confidence

### Factors That Decrease Confidence:
- Test code detection
- Comments
- Example patterns

### Example Output:
```json
{
  "id": "file.ts:42-SQLInjection",
  "severity": "critical",
  "title": "SQL Injection",
  "confidence": 88,
  "file": "src/db.ts",
  "line": 42,
  "code": "await db.execute(`SELECT * FROM users WHERE id = ${id}`)"
}
```

## 7. Severity & Scoring

### Severity Levels:
- CRITICAL (50 points): Immediate action required
- HIGH (20 points): Fix soon  
- MEDIUM (5 points): Consider fixing
- LOW (1 points): Nice to fix

### Score Calculation:
```
Deductions = sum of severity weights per UNIQUE file
Score = max(0, 100 - deductions)
```

### Grade Thresholds:
- A (90-100): Excellent
- B (75-89): Good
- C (60-74): Needs work
- D (40-59): Poor
- F (<40): Critical issues

## 8. API Endpoints

### POST /api/jobs
Create a new async scan job.

Request:
```json
{
  "repoUrl": "https://github.com/owner/repo",
  "token": "ghp_xxx"  // optional - for higher rate limits
}
```

Response (202):
```json
{
  "jobId": "uuid-v4-formatted",
  "status": "queued",
  "message": "Scan started. Use GET /api/jobs?jobId=<id>"
}
```

### GET /api/jobs?jobId=<jobId>
Check scan job status and get results.

Response:
```json
{
  "id": "uuid-v4-formatted",
  "repoUrl": "https://github.com/owner/repo",
  "status": "completed",
  "score": 75,
  "grade": "B",
  "confidence": 85,
  "filesScanned": 42,
  "vulnerabilities": [
    {
      "id": "file.ts:42-Injection",
```

## 9. Vulnerability Detection (50+ Patterns)

### Secrets Detection (CRITICAL)
- OpenAI API keys (sk-...)
- xAI API keys
- GitHub tokens (ghp_, gho_, ghs_)
- AWS keys (AKIA...)
- JWT tokens
- Stripe keys (sk_live_)
- Auth0 secrets

### SQL/NoSQL Injection (CRITICAL)
- Template string interpolation
- f-string in SQL
- Execute with string concatenation
- MongoDB $where operator

### Code Injection (CRITICAL)
- eval() usage
- exec() usage
- pickle.loads()
- __import__("os")

### XSS Vulnerabilities (HIGH)
- innerHTML assignment
- dangerouslySetInnerHTML
- document.write()

### Crypto Issues (HIGH)
- MD5 hashing
- SHA1 hashing
- Math.random() for security
- Weak bcrypt iterations

### Command Injection (HIGH)
- subprocess with shell=True
- os.system()
- os.popen()

### Configuration Issues (HIGH)
- CORS wildcard (*)
- DEBUG mode enabled
- Verbose error messages

### SSRF (CRITICAL)
- URL built from user input
- fetch/axios with string concatenation

### Logging Issues (HIGH)
- Credentials in console.log
- Passwords in error logs

## 11. Known Limitations

1. **No taint tracking** - Basic regex patterns only
2. **No cross-file analysis** - Can't track data flow
3. **GitHub API rate limits** - 60/hour unauthenticated
4. **In-memory storage** - Jobs lost on restart (Supabase planned)
5. **No AST parsing** - Regex vs proper AST analysis
6. **False positives possible** - Confidence scoring helps filter
